# Канон CDM: Операционный уровень минимальной достаточности

## 1. Основное положение

**Операционный уровень системы должен использовать минимально
достаточную тринарную шкалу результатов:**

\[ Result `\in `{=tex}{+1,;0,;-1} \]

где:

-   **+1** --- цель достигнута / изменение согласовано с Intent;
-   **−1** --- цель не достигнута / изменение приводит к деградации;
-   **0** --- недостаточно оснований для бинарного решения.

Ключевой принцип:

> **0 --- это не тип неопределённости, а операционный сигнал
> недостаточной определённости для действия.**

Все причины неопределённости агрегируются в состояние 0 на уровне
выполнения.

------------------------------------------------------------------------

## 2. Декомпозиция неопределённости

Аналитически состояние 0 может возникать по различным причинам:

  -----------------------------------------------------------------------
  Класс               Причина                   Природа
  ------------------- ------------------------- -------------------------
  A                   Нет данных                эпистемическая

  B                   Шум / вероятность         стохастическая

  C                   Неприменимость            семантическая
                      (семантическая            
                      несовместимость           
                      контекстов)               
  -----------------------------------------------------------------------

Однако в операционном опыте эти причины **не различаются**.

Декомпозиция выполняется только на аналитическом уровне (evaluate /
пост‑анализ).

------------------------------------------------------------------------

## 3. Связь с ChangeFlow

В ChangeFlow результат формируется после реализации воздействия
(implement) и фиксируется в evaluate:

-   implement → Result (+1 / 0 / −1)
-   evaluate → интерпретация результата

Состояние 0 выполняет три функции:

1.  остановка или отложение действия;
2.  запуск анализа;
3.  сигнал обучения.

------------------------------------------------------------------------

## 4. Связь с CtxL (Context‑Dependent Logic)

Семантическая неприменимость (CtxL‑состояние M) является частным случаем
операционного 0:

\[ M `\subset 0`{=tex} \]

Но не наоборот:

\[ 0 `\not`{=tex}`\subset `{=tex}M \]

Таким образом, 0 --- супермножество всех форм неопределённости
выполнения.

------------------------------------------------------------------------

## 5. Архитектурное следствие

Операционные системы должны:

-   хранить только тринарный результат (+1 / 0 / −1);
-   выполнять классификацию причин 0 на аналитическом уровне;
-   не усложнять runtime‑логику типами неопределённости.

Это обеспечивает:

-   минимальную вычислительную сложность;
-   универсальность обучения;
-   доменную независимость модели опыта.

------------------------------------------------------------------------

## 6. Фундаментальное следствие

Тринарная шкала соответствует универсальному механизму обучения систем:

-   +1 --- закрепление стратегии;
-   −1 --- подавление стратегии;
-   0 --- исследование и поиск новых решений.

Таким образом:

> **0 является основным источником адаптации и эволюции поведения
> системы.**

------------------------------------------------------------------------

## 7. Каноническая формулировка

> В операционном опыте все формы неопределённости агрегируются в
> состояние 0.\
> Декомпозиция причин состояния 0 относится к аналитическому уровню и не
> требуется для выполнения действия.

------------------------------------------------------------------------

## 8. Статус

Канон совместим с:

-   Intent
-   ChangeFlow
-   Experience
-   CtxL
-   DegradeScore

Документ предназначен для включения в канонический корпус CDM.
