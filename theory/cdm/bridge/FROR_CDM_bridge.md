# FROR <-> CDM Bridge: каноническое соответствие

------------------------------------------------------------------------

## 1. Назначение

Документ фиксирует, как CDM используется как канонический операционный и онтологический слой для
реализации инвариантов FROR.

Коротко:
- `FROR` задает инварианты и ограничения;
- `CDM` задает каноническую логику применимости, выбор, commit и оценку результата.

------------------------------------------------------------------------

## 2. Принцип интеграции

Интеграция считается корректной, если CDM-механизмы не подменяют FROR-инварианты,
а реализуют их в исполнимом процессе.

`FROR` отвечает на вопрос: что обязано сохраняться.
`CDM` отвечает на вопрос: как это исполняется в каноническом контекстно-зависимом цикле.

------------------------------------------------------------------------

## 3. Каноническое отображение FROR -> CDM

| FROR | CDM слой | Смысл |
|---|---|---|
| `Ω` | пространство состояний под веткой `k` и активным `C_active^k` | домен потенциальных микросостояний |
| `Σ` | фиксируемые runtime-репрезентации результата/контекста | ограниченное множество различимых состояний |
| `F: Ω->Σ` | процедура интерпретации/классификации в активном контексте | акт фиксации различия |
| `c(σ,σ')` | операционная цена перехода и записи | стоимость изменения фиксированного различия |
| `τ` | накопленная цена commit-цепочки | скаляр необратимости |
| `{-,0,+}` | `Result in {+1,0,-1}` + `fror_zero_class` (аналитика) | тринарная классификация сохраняется, но `0` разделяется: `0_structural -> Result=0`, `0_no_cost -> аналитическая метка` |
| `G` | граф допустимых переходов процесса | микродинамика |
| `G⁰` | переходы без роста `τ` в выбранной нормализации | нулевая геометрия альтернатив |
| `SLOT` | локально устойчивые контекстно-согласованные режимы | аттракторы/устойчивые структуры |

------------------------------------------------------------------------

## 4. Связь с фазами ChangeFlow

Рабочее соответствие (операционный профиль):
- `CF1-CF4`: подготовка выбора (контекст, применимость, варианты);
- `CF5`: commit (необратимая фиксация на уровне процесса);
- `CF6`: feedback и обновление метрик.

FROR-интерпретация:
- commit увеличивает `τ` при ненулевой цене;
- rollback после commit трактуется по канону FROR I3--I4
  ([FROR_normalizations_invariants](../../fror/FROR_normalizations_invariants.md))
  как новый согласованный переход (компенсация), а не как полная
  инверсия истории.

------------------------------------------------------------------------

## 5. Инварианты, которые CDM-реализация обязана сохранять

### B1. Конечность различения

CDM-профиль должен явно задавать конечное множество фиксируемых runtime-состояний
или его эффективную верхнюю границу.

### B2. Небесплатность нетривиальной фиксации

Любой нетривиальный commit/переключение состояния должен иметь положительную
операционную цену в выбранной нормализации.

### B3. Явный режим трихотомии

`{-,0,+}` задается как:
- `A3-mode` (аксиоматика FROR), либо
- `Derived-mode` (из оператора применимости/дефекта).

### B4. Необратимость как ресурсное ограничение

Bridge использует каноническую формулировку необратимости из FROR I3--I4
([FROR_normalizations_invariants](../../fror/FROR_normalizations_invariants.md)):
обратный путь не запрещен, но при конечных ресурсах точная инверсия обычно
вычислительно-энергетически нереализуема относительно forward-компенсации.

------------------------------------------------------------------------

## 6. Минимальный DSL-контракт FROR-совместимого CDM профиля

```yaml
profile: "fror_cdm_min"
mode_trichotomy: "A3-mode"   # или Derived-mode

runtime:
  states_sigma: "<finite_or_bounded>"
  fixation_operator: "F"
  transition_cost: "c"
  tau_accumulator: true

result_mapping:
  plus: "positive_change"
  zero: "applicability_failure_only"
  minus: "negative_or_relaxation"

commit_policy:
  irreversible_commit_point: "CF5"
  rollback_semantics: "compensation_forward"
```

------------------------------------------------------------------------

## 6.1 ResultMappingPolicy (нормативно для CDM-совместимости)

Для совместимости с каноническим CDM принимается:

1. `CDM.Result = 0` **только если**
   `ApplicabilityFailure(Intent, C_active^k, LC_phase)`.
2. `FROR.0_structural` (контекст/фаза не допускают реализацию) отображается в
   `CDM.Result = 0`.
3. `FROR.0_no_cost_transition` (переход без роста `tau`) **не** отображается в
   `CDM.Result = 0`; он фиксируется как:
   - аналитическая метка (`fror_zero_class=no_cost_transition`) при `CDM.Result in {+1,-1}`, или
   - отдельный служебный non-result флаг.

Формально:

`CDM.Result=0 <=> ApplicabilityFailure(Intent, C_active^k, LC_phase)`.

`FROR.no_cost_transition => (tau_delta=0)` и `CDM.Result != 0`.

Это исключает смешение семантики неприменимости и семантики нулевой цены перехода.

------------------------------------------------------------------------

## 7. Фальсифицируемые критерии совместимости

Связка `FROR <-> CDM` считается опровергнутой для домена, если наблюдается
хотя бы одно:

1. устойчивое изменение без цены (`c=0` для нетривиальных переходов);
2. отсутствие эффективной конечности `Σ` при сохранении рабочей модели;
3. невозможность воспроизводимо сопоставить:
   - runtime-исходы с `CDM.Result in {+1,0,-1}` (где `0` только для ApplicabilityFailure),
   - аналитические FROR-классы (`{-,0,+}`) с mapping-политикой раздела 6.1;
4. rollback, систематически восстанавливающий полную историю дешевле/проще,
   чем forward-компенсация при сопоставимых условиях.

------------------------------------------------------------------------

## 8. Итог

`CDM` является канонической рамкой интеграции FROR, если
используется как слой исполнения, а не как замена инвариантов.

Формула интеграции:

`FROR (инварианты) + CDM (каноническая логика) = проверяемая междоменная рамка`.
