# Спецификация: System (каноническая)

## 1. Область определения

Документ задает каноническое определение `System` в CDM и границы его связи с:
- `LifeCycle-6`;
- `ChangeFlow-6`;
- `Context` / `C-Layer`;
- `Identity`.
- `Subjectivity`.

Канон фиксирует инварианты сущности системы. Доменные классификации и шкалы сложности выносятся в отдельные профили.

---

## 2. Нормативные ссылки

- `LifeCycle-6 (каноническая)`
- `ChangeFlow-6 (каноническая)`
- `Context (каноническая)`
- `Intent (каноническая)`
- `CtxL (каноническая)`
- `Identity (каноническая)`
- `Subjectivity (каноническая)`

---

## 3. Каноническое определение

Система `Y` в CDM — это идентифицируемая сущность, для которой одновременно определимы:

1. собственный режим существования через `LifeCycle` (`LC_Y`);
2. собственные акты изменения через `ChangeFlow` (`CF_Y`);
3. пространство контекстных ограничений (`C-Layer(Y)`), влияющих на применимость изменений;
4. наблюдаемая причинная связность между изменениями состояния и результатом (`Experience`).

Формально (каноническая форма):

`System(Y) => <Identity(Y), LC_Y, CF_Y, C-Layer(Y), Experience_Y>`.

---

## 4. Минимальные инварианты системы

Для сущности `Y`, признанной системой, должны выполняться:

1. **I1: Идентифицируемость**  
   Существует наблюдательно воспроизводимый критерий идентичности `Identity(Y)` в пределах текущего LC.

2. **I2: LC-определимость**  
   Существует допустимое описание `LC_Y` в терминах канонических фаз `F1..F6`.

3. **I3: CF-определимость**  
   Для как минимум одного изменения `Y` существует представление через `CF1..CF6` (или допустимую редукцию для низких классов).

4. **I4: Контекстная обусловленность**  
   Существует непустой `C-Layer(Y)`, и применимость `Intent` определяется относительно `C_active`.

5. **I5: Наблюдаемая результативность**  
   Изменения `Y` допускают классификацию результата в каноническом операционном домене (`+1/0/-1`) с канонической семантикой неприменимости.

---

## 5. Граница канона и профилей

Канон `System`:
- не задает доменных классов сложности;
- не фиксирует конкретные метрики интеллекта/агентности;
- не требует конкретного механизма `decide` как универсального критерия системности.

Профильные документы могут вводить:
- классы систем;
- эвристики декомпозиции;
- операционные признаки автономности;
при условии, что они не противоречат инвариантам §4.

---

## 6. Связь с LifeCycle и ChangeFlow

1. `LifeCycle` задает режим существования системы.
2. `ChangeFlow` задает морфологию реализации конкретного изменения.
3. `CF5` остается единственной точкой физической фиксации изменения в рамках ChangeFlow.
4. `CF6` остается единственной точкой обновления метрик LC.

---

## 7. Связь с Context

`System(Y)` валидна только при явной внешней контекстной обусловленности:

- `C-Layer(Y) != ∅`;
- `C_active(Y,t) in C-Layer(Y)`;
- `not Applicable(Intent, C_active, LC_phase) => Experience=0`.

Контекст может быть системой на собственном уровне анализа, но в отношении `Y` рассматривается как внешний фактор.

---

## 8. Фальсифицируемость

Определение `System` считается неприменимым к классу сущностей, если при достаточной наблюдаемости устойчиво не удается установить хотя бы один из инвариантов I1-I5.

Недостаточная наблюдаемость не считается фальсификацией.

---

## 9. Заключение

Каноническое определение `System` в CDM фиксирует минимальный инвариантный каркас:

`Identity + LifeCycle + ChangeFlow + Context + Experience`.

Все доменные и прикладные классификации строятся как профили поверх этого каркаса.
