# Профиль: System Classification (прикладной)

## 1. Назначение

Документ задает прикладной профиль классификации систем и декомпозиции поверх канонического определения `System`.

Профиль предназначен для:
- инженерной типизации систем;
- прикладной декомпозиции на подсистемы/процессы;
- согласования правил `RLC/CC`, `ROS`, `SEC/OC`.

---

## 2. Нормативная граница

Это **профиль**, а не канон.

Источник канона:
- `System (каноническая)`
- `LifeCycle-6 (каноническая)`
- `ChangeFlow-6 (каноническая)`
- `Context (каноническая)`
- `Intent/CtxL (канонические)`

При конфликте приоритет у канонических документов CDM.

---

## 3. Профильные классы систем (операционный слой)

Классы являются прикладной шкалой сложности, а не онтологическим ядром канона.

Примерная шкала `C0..C6`:

1. `C0` — детерминированные системы с редуцированным локальным изменением.
2. `C1` — естественные системы с полным CF-представлением на уровне наблюдения.
3. `C2` — автономные локально-целеориентированные системы (ограниченная адаптивность).
4. `C3` — коллективные системы с эмерджентной координацией.
5. `C4` — индивидуальные когнитивные системы (уровень 1 рефлексивности).
6. `C5` — индивидуальные когнитивные системы (уровень 2 рефлексивности).
7. `C6` — метасистемы/сверхсистемы с вложенными контурами координации.

Численные критерии переходов между классами задаются отдельной доменной политикой.

---

## 4. Профильная декомпозиция: subsystem vs process

Для объекта `X` относительно родителя `S` профиль допускает три прикладных режима:

1. `Subsystem(S)`  
   Если у `X` различимы собственные инварианты системы (канон `System`) и отдельный LC-контур на выбранном уровне описания.

2. `DeterministicSubsystem(S)`  
   Частный случай `Subsystem(S)` с предсказуемой динамикой выбора в контуре наблюдения.

3. `ProcessOf(S)`  
   Если `X` не удовлетворяет критериям отдельной системной идентичности и описывается как процесс изменения внутри `S`.

Важно:
- наличие/отсутствие конкретного механизма `decide` не является единственным универсальным критерием системности;
- `decide` может использоваться как один из операционных индикаторов в доменном профиле.

---

## 5. Связь с RLC/CC, ROS, SEC/OC

Профиль задает опорные определения для будущей переработки прикладных правил:

1. `RLC/CC` ([[fcdm-core/theory/cdm/Specifications/AppliedRules/RLC-CC-Profile|RLC-CC-Profile]])  
   Используется как фильтр различения системной декомпозиции и процессной декомпозиции.

2. `ROS` ([[fcdm-core/theory/cdm/Specifications/AppliedRules/ROS-Profile|ROS-Profile]])  
   Используется как правило остановки декомпозиции после прохождения `RLC/CC`.

3. `SEC/OC` ([[fcdm-core/theory/cdm/Specifications/AppliedRules/SEC-OC-Profile|SEC-OC-Profile]])  
   Используется как фильтр внешнего контекста относительно текущего `Y`.

4. `Observer` ([[fcdm-core/theory/cdm/Specifications/AppliedRules/Observer-Profile|Observer-Profile]])  
   Используется для ресурсно-ограниченной фиксации различий и координации контекстов.

Порядок применения в прикладной практике:

`RLC/CC -> ROS -> SEC/OC` (или `RLC/CC -> SEC/OC -> ROS` в зависимости от контура анализа),  
при неизменном соблюдении канонических ограничений CDM.

---

## 6. Контекстные роли (профиль)

В прикладной модели вводятся ролевые типы контекста:

1. `C_obs`  
   Контур фиксации различий при ограниченных ресурсах наблюдения.

2. `C_coord`  
   Контур координации правил/ограничений/синхронизации.

3. `C_meta`  
   Метаконтур управления контекстами: выбор активных контуров, приоритеты, политика разрешения конфликтов, распределение ресурсов.

Статус:
- `C_obs_coord` как отдельный тип контекста в этом профиле **deprecated**;
- legacy-термин `C_obs_coord` допускается только как read-only alias
  для композиции `C_obs + C_coord` под `C_meta`;
- если один узел исполняет обе функции, это моделируется композицией ролей `C_obs + C_coord` под политикой `C_meta`.

---

## 7. Минимальный checklist профиля

Перед применением класса/декомпозиции проверить:

1. Канонические инварианты `System` выполнены.
2. Уровень анализа (ROS) зафиксирован.
3. Контекстная граница (`SEC/OC`) установлена.
4. Роли контекста назначены явно (`C_obs`, `C_coord`, `C_meta`).
5. Критерии классификации заданы явно и воспроизводимы.
6. Решение о типе (`Subsystem`, `DeterministicSubsystem`, `ProcessOf`) трассируемо.

---

## 8. Ограничения профиля

Профиль не должен:
- переопределять канонические фазы `F1..F6` и `CF1..CF6`;
- переопределять каноническую семантику `Result/Experience`;
- вводить обязательные универсальные критерии, отсутствующие в каноне.

---

## 9. Заключение

Профиль классификации систем предназначен для прикладной инженерной типизации и декомпозиции, оставаясь совместимым с каноническим каркасом CDM.

Он подготавливает основу для актуализированной интеграции документов `RLC/CC`, `ROS`, `SEC/OC` без возврата к устаревшей decide-центричной формулировке.
