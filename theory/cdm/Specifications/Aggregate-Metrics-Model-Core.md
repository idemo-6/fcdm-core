# Модель: Aggregate Metrics (core)

## 2.2.12.1. Основное

Документ задает универсальную рамку агрегатных метрик: шаблон уровня 0,
в который любой домен подставляет собственные скаляры и получает
сопоставимую проекцию в `DegradeVector` и `DegradeScore (DS)`.

Версия `2.2.12` портирована в CDM model-core слой с сохранением структуры.

Нормативная база:
- [[fcdm-core/theory/cdm/Specifications/DegradeVector-Canonical|DegradeVector-Canonical]]
- [[fcdm-core/theory/cdm/Specifications/LifeCycle-6_v2|LifeCycle-6]]
- [[fcdm-core/theory/cdm/Specifications/CF-LC-Evaluate|CF-LC-Evaluate]]
- [[fcdm-core/theory/cdm/Specifications/DegradeVector-Machine-Contract|DegradeVector-Machine-Contract]]
- [[fcdm-core/theory/cdm/Specifications/DegradeVector-Domain-Template|DegradeVector-Domain-Template]]

Нотация фаз:
- каноническая: `CF1..CF6`;
- доменная (алиасы): `collect/analyze/forecast/decide/implement/evaluate`.

---

## 2.2.12.3. Назначение

Дать единообразный шаблон, куда любой домен вставляет собственные скаляры и
сразу видит, как они ложатся в `DegradeVector`, `DS` и фазовую логику `LifeCycle-6`.

Цель:
- убрать "табличную магию";
- сделать рамку машиночитаемой;
- обеспечить peer-review и воспроизводимость.

## 2.2.12.4. DegradeScore (DS): агрегатная операционная метрика

`DS` — агрегатная скалярная операционная метрика состояния системы в контуре
`LifeCycle-6`/`ChangeFlow-6`.

`DS` не описывает полную внутреннюю сложность системы, а дает однозначный
операционный сигнал для фазовой логики.

### 2.2.12.4.1. Принцип агрегации

Реальная система содержит множество доменных метрик `rr_i`, которые могут быть
разнородными по природе и частоте измерений.

Примеры:
- физические (износ, температура, шум);
- логические (целостность, согласованность);
- экономические (стоимость, рентабельность);
- организационные (нагрузка, устойчивость решений).

`DS` является агрегированной сверткой, а не заменой этих метрик.

### 2.2.12.4.2. Операционная редукция сложности

Для аналитики разрешен отдельный агрегатный знак воздействия:
- `op_signal = -1` — ожидаемо ухудшение (`Delta_DS > 0`),
- `op_signal = 0` — ожидаемо близко к нейтрали (`Delta_DS ~= 0`),
- `op_signal = +1` — ожидаемо улучшение (`Delta_DS < 0`).

Важно:
- `op_signal` не является runtime `Result`;
- runtime `Result in {+1,0,-1}` остается каноническим и
  `Result=0 <=> ApplicabilityFailure`.

### 2.2.12.4.3. Формирование DegradeScore

Базовый шаблон:

`DS = sigma(sum_i(omega_i * rr_i))`, `DS in [0,1]`

где:
- `rr_i` — нормализованные доменные метрики,
- `omega_i` — веса,
- `sigma` — допустимая функция сглаживания/нормализации.

Эквивалентные свертки допустимы, если они открыты и воспроизводимы.

Каноническое направление шкалы:
- `DS = 0` — лучший режим,
- `DS = 1` — худший режим.

### 2.2.12.4.4. Интерпретация DS как вычислительной энтропии

`DS` может интерпретироваться как вычислительная энтропия контура изменений:
- рост дефекта и цены коррекции,
- рост необратимости пути изменения,
- снижение управляемости.

Это вычислительная, а не термодинамическая энтропия.

### 2.2.12.4.5. Связь с LifeCycle-6

`DS` используется в фазовых ограничениях `LifeCycle` через `DegradeVector` и
временные условия (`tau_stable_min`, `tau_phase_min`):
- низкие `DS` соответствуют устойчивым режимам,
- устойчивый рост `DS` соответствует деградации,
- достижение `DS = 1` связано с предельным режимом.

### 2.2.12.4.6. Аналитическое раскрытие

При входе в риск-зону (`DS -> 1`) рекомендуется запуск полного аналитического
раскрытия (для классов систем `>= 2`):

`rr_i, omega_i -> DS in [0,1] -> op_signal in {-1,0,+1}`

с декомпозицией:
- вкладов `rr_i`,
- весов `omega_i`,
- устойчивости mapping `op_signal -> Delta_DS`.

Для классов `0-1` допускается редуцированный режим без развитой аналитики.

### 2.2.12.4.7. Итоговое положение

`DS` — операционный интерфейс между:
- многомерной сложностью системы,
- фазовой топологией `LifeCycle-6`,
- исполнительским контуром `ChangeFlow-6`.

Сложность не исчезает, а сворачивается в управляемую форму.

## 2.2.12.5. Реестр доменных доноров (плейсхолдеры)

| ID-донора | Домен | Физический смысл | Базовая метрика (до нормализации) | Нормализация к деградации |
|---|---|---|---|---|
| `DS_BIO` | биология | Доля нежизнеспособных элементов | `dead / total` | `0 = все живы`, `1 = все мертвы` |
| `DS_STAR` | звезды | Доля выработанного топлива | `1 - (H_now / H_init)` | `0 = старт`, `1 = исчерпано` |
| `DS_CODE` | ПО | Доля неуспешных прогонов | `1 - (success / total)` | `0 = все зеленые`, `1 = все красные` |
| `DS_CORP` | компании | Доля потерь штата | `leavers / headcount` | `0 = никто не ушел`, `1 = ушли все` |
| `DS_TOKEN` | крипто-экономика | Доля неактивных валидаторов | `1 - (active / total)` | `0 = все online`, `1 = все offline` |
| `DS_CITY` | городское | Доля неработающих узлов | `down / (up + down)` | `0 = все работает`, `1 = все не работает` |

Правило добавления нового донора:
1. Метрика однозначно нормализуется в `[0,1]`.
2. Есть публичный способ измерения.
3. Указаны единицы времени для `tau_stable_min` и `tau_phase_min`.

## 2.2.12.7. Шаблон доменного отчета (YAML)

```yaml
domain: "stellar_physics"
system: "Sun"
class_tag: "C-1-Natural"
donor_id: "DS_STAR"
DS_formula: "1 - (H_mass_now / H_mass_ZAMS)"
tau_stable_min: 1e6   # years
tau_phase_min: 1e5    # years
source_data: "https://doi.org/10.0000/stellar_ds_2025"
code: "https://github.com/xxx/ds-sun"

analysis:
  op_signal: -1
  expected_delta_ds_sign: positive

runtime:
  result: 1
  note: "result=0 reserved only for ApplicabilityFailure"
```

---

## 2.2.12.8. Как проверить совместимость с другим доменом

1. Взять два YAML-файла из разных доменов.
2. Привести единицы времени к общему масштабу.
3. Сравнить форму `DS(t)` на нормализованных шкалах.
4. Если формы расходятся, это исследовательский вопрос, а не автоматическая
   ошибка домена.

---

## 2.2.12.9. Итоговое правило

Каждая строка таблицы порогов должна содержать плейсхолдеры и типы данных.
Численные значения считаются валидными только при публикации датасета и кода.

---

## 2.2.12.10. Заключение

Агрегатные метрики задаются как машиночитаемая универсальная рамка,
которая сохраняет сопоставимость между доменами и совместима с каноном CDM.
