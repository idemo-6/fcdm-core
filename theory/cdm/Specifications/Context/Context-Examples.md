# Примеры: Context в CDM

## 1. Техническая система (сервис)

Система `Y`: backend-сервис.

`C-Layer(Y)`:
- `C_phys`: вычислительные ресурсы хоста;
- `C_legal`: регуляторные ограничения по данным;
- `C_info`: входной поток запросов и телеметрия;
- `C_org`: SLA и приоритеты продукта.

`C_candidates` для Intent "уменьшить latency p95":
- `C_phys`, `C_info`, `C_org`.

Пример последовательной обработки:
- шаг 1: `C_active = C_info` (диагностика профиля нагрузки);
- шаг 2: `C_active = C_phys` (проверка ресурсных ограничений);
- шаг 3: `C_active = C_org` (проверка SLA-приоритетов).

`OperatorSet(C_active, LC_phase)`:
- профилирование;
- изменение конфигурации кэша;
- перераспределение ресурсов.

Если регуляторное ограничение запрещает хранение нужных данных профиля и других операторов недостаточно для реализации Intent:
- `Applicable = false`;
- `Experience = 0`.

---

## 2. Организационная система (проект)

Система `Y`: продуктовая команда.

`C-Layer(Y)`:
- `C_eco`: бюджет;
- `C_legal`: контрактные обязательства;
- `C_org`: внутренняя структура ролей;
- `C_info`: пользовательская обратная связь.

`C_candidates` для Intent "ускорить выпуск версии":
- `C_eco`, `C_org`, `C_legal`.

Пример переключения:
- шаг 1: `C_active = C_org` (оценка загрузки команды);
- шаг 2: `C_active = C_eco` (оценка бюджетного ограничения);
- шаг 3: `C_active = C_legal` (проверка контрактных обязательств).

Если бюджет и контрактные сроки конфликтуют, а политика приоритетов не разрешает конфликт:
- `Applicable = false`;
- `Experience = 0` на Intent-уровне.
