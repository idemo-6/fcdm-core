# Спецификация: Context (каноническая)

## 1. Область определения

Документ задает каноническое определение `Context` и `Multicontext` в CDM.

Канон фиксирует только инварианты:
- что считается контекстом;
- как формируется `C-Layer`;
- как выделяется `C_active`;
- как контекст ограничивает реализуемость `Intent` через `CF1..CF6`.

Доменные типы контекста, математические модели и примеры вынесены в отдельные документы.

## 1.1 Базовые принципы мультиконтекстности

Канон Context в CDM опирается на три базовых принципа:

1. `System-first`: сначала фиксируется единая система `Y`, затем для нее выделяются контексты.
2. `Domain-as-context`: доменная перспектива трактуется как контекстная проекция одной системы, а не как отдельная онтология системы.
3. `Cross-context semantics`: межконтекстные взаимодействия являются частью модели, а не внешним комментарием.

Следствие:
- различие между `Undefined` и `Inapplicable` обязательно сохраняется;
- `Undefined` рассматривается как частный случай `Inapplicable` в рамках фиксированного контекста;
- определенность высказывания в одном контексте не гарантирует его применимость в другом.

---

## 2. Каноническое определение

Для системы `Y` контекстом `C` считается сущность `X`, если одновременно выполняется:

1. `X != Y`;
2. `X` не является частью внутренней декомпозиции `Y` на выбранном уровне ROS;
3. изменение `X` может изменить допустимое пространство состояний или допустимые траектории изменения `Y`.

Канонически:

`Context(Y) := { X | External(X, Y) and Influence(X -> Y) }`.

---

## 3. C-Layer, Active Context и мультиконтекстность

`C-Layer(Y)` — множество всех контекстов системы `Y`:

`C-Layer(Y) = { C1, C2, ... , Cn }`, `n >= 1`.

В любой момент времени в рамках одной исполнительной ветки активен ровно один контекст:

`C_active(Y, t) in C-Layer(Y)`.

Для задачи может существовать множество контекстов, которые требуется согласовать:

`C_candidates(Y, t) subseteq C-Layer(Y)`, `|C_candidates| >= 1`.

`Multicontext` — это режим задачи, в котором `|C_candidates| >= 2`.
Он реализуется через переключение активного контекста во времени или через несколько параллельных веток, где в каждой ветке активен один контекст.

---

## 4. Active Context

`C_active(Y, t)` — единственный активный контекст из `C-Layer(Y)`, релевантный в момент `t` для текущего `Intent` и фазы `LifeCycle`.

Контекст активен, если его ограничения, ресурсы или операторы реально участвуют в текущем ChangeFlow.

Если для задачи требуется межконтекстная согласованность, активный контекст последовательно переключается между элементами `C_candidates`, либо обрабатывается в параллельных ветках с последующей координацией.

---

## 5. Связь с ChangeFlow и Intent

Контекст не является фазой `ChangeFlow`, но ограничивает каждую фазу `CF1..CF6`.

`Intent` проверяется на применимость только относительно текущего `C_active`.

Минимальный канонический принцип:

`not Applicable(Intent, C_active, LC_phase) => Experience = 0`.

Интерпретация `Experience` задается спецификацией Intent и спецификацией Evaluate.

---

## 6. Иерархия доступности контекстов (LC и Parent -> Child)

`LifeCycle` задает локальный фазовый контур доступности контекстов для системы `Y`.

Канонически:

`C_avail(Y, LC_phase) subseteq C-Layer(Y)`.

`C_candidates(Y, t) subseteq C_avail(Y, LC_phase(t))`.

Для иерархии систем (`Parent -> Child`) доступность контекстов дочерней системы ограничена политикой родительской системы:

`C_avail_child(LC_phase_child) subseteq C_avail_parent(LC_phase_parent, policy_parent)`.

Тем самым:
- `LC` выполняет локальную gatekeeping-функцию доступности контекстов;
- `C_coord` выполняет функцию разрешения конфликтов между уже доступными контекстами;
- `C_meta` выполняет функцию управления активацией, приоритетами и политикой выбора между контекстами.

---

## 7. Граница с Change Operators

`Change Operators` не входят в определение `Intent`.

`OperatorSet` вычисляется для активного контекста и текущей фазы `LifeCycle`:

`OperatorSet = OperatorSet(C_active, LC_phase)`.

Реализация `Intent` выбирает операторы только из доступного `OperatorSet(C_active, LC_phase)`.

---

## 8. Декларация, использование и материализация контекста

В профиле `ICSS-IntentFlow v1` фиксируются три разные операции:

1. `declare(ctx)` — объявление контекста в collect/declaration блоке (`@(... )`);
2. `use(ctx)` — чистое логическое использование контекста (без I/O);
3. `materialize(ctx)` — внешняя материализация контекста только через `pt.*`.

Каноническое правило:

`materialize(ctx) => ctx in DeclaredCtxSet`.

Нарушение соответствия declared/materialized контекстов является ошибкой валидации профиля.

---

## 9. Политика неантропоморфности

В каноне допускаются только обозначения:
- `Context`, `C-Layer`, `C_active`;
- `LifeCycle`, `CF1..CF6`, `Intent`, `Experience`;
- `OperatorSet`.

Доменные названия (например, `collect/analyze/...`) допускаются только через `DomainLexicon`.

---

## 10. Фальсифицируемость

Канон Context опровергается для класса систем, если показано хотя бы одно из:

1. есть внешнее влияние на `Y`, которое не может быть представлено через `Context`;
2. есть реализуемый `Intent`, для которого контекстные ограничения не влияют на `CF1..CF6`;
3. `C_active` не позволяет объяснить наблюдаемое `Experience`.
